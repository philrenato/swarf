v00000-011 r12 — TOOLPATH OFFSET FIX DETAIL (2026-04-16)
──────────────────────────────────────────────────
FIXED "toolpath offset bug" — Priority 1 blocker from r10-final.

THREE ROOT CAUSES IDENTIFIED AND FIXED:

1. Non-centered cube.stl. The seed cube (web/obj/cube.stl) was the
   upstream Kiri promotional cube: 320 triangles, bounds at X[50,75]
   Y[110,135] Z[21,46] — NOT centered at origin. Kiri's widget.center()
   normalizes vertices after load, but the raw bounds were transiently
   visible to platformUpdateStock, producing a stock.center of
   (62.5, 122.5) before centering completed. Replaced with a proper
   12-triangle 25.4mm cube: X[-12.7, 12.7] Y[-12.7, 12.7] Z[0, 25.4].

2. Auto-inject stock padding lost between slice and prepare. The slice
   worker's auto-inject (camStockX/Y/Z = 5mm) modified settings.process,
   but the prepare dispatch receives a FRESH settings copy from the main
   thread via postMessage (worker.js line 532). The 5mm padding was
   invisible to prepare, anim-2d, and export.
   Fix: slice.js now stashes padded stock dimensions on the widget
   object (widget._swarfPaddedStock). prepare.js checks for this and
   uses it when present, ensuring prepare uses the same stock envelope
   the slicer used to generate paths.

3. Dual position sources in prepare.js. printPoint was initialized from
   settings.stock.center (main-thread bounds midpoint), while wmx/wmy
   came from widget.track.pos. These can diverge in multi-widget or
   stored-profile scenarios. Added a defensive fallback: if center is
   missing/undefined, derive it from widget.track.pos instead.

v00000-010 r10 — FLOOR-STABLE-ACROSS-MODES (2026-04-15)
────────────────────────────────────────────────────────
FIXED the "floor brighter during SIMULATE" bug flagged at end of r9.
Root cause: Kiri's Space.scene.updateFog() (src/moto/space.js:1489) sets
fog.near = camera.distanceTo(target), fog.far = near × mult, and re-fires
on every camera move (space.js:1343). Switching to SIMULATE shifts the
camera, which shifts the fog band in world space, which paints the grid
brighter or darker depending on direction. None of the r9 hypotheses
(tonemapping, environment, hemi intensity, grid envMap) were the cause.

Fix in web/kiri/swarf-grid-fade.js:
  - fog.near / fog.far now computed from part bbox in WORLD UNITS
    (near = max(size×4, 120), far = max(size×18, near×4)) and written
    directly onto the THREE.Scene fog object.
  - Space.scene.updateFog is monkey-patched to a no-op so Kiri's camera-
    relative recompute can't clobber the pinned values.
  - Removed 'animate' and 'function.animate' listeners — fog now only
    updates on widget.add / widget.delete.

Verified via tools/swarf_probe.mjs --flow=sim --shot: PREVIEW and
SIMULATE screenshots show matching floor gradient and grid fade.

Two non-obvious truths, either one of which will burn hours if you
don't know them:

1. **Non-indexed CAM uses anim-2d.js, NOT anim-3d.js.** Phil's mill
   workflow (cube + rough op on the bed, no A-axis indexing) runs
   through `env.animVer = 0` → `anim-2d.js`. Kiri's built-in lineTracker
   (the red additive path-draw) lives only in `anim-3d.js` and only
   runs when env.isIndexed → env.animVer = 1. So any edits to
   `src/kiri/mode/cam/app/anim-3d.js` to "fix the ribbon" will compile,
   land, and do exactly nothing for Phil's use case. The ribbon HAS
   to be driven from the `swarf.tool.move` event stream that BOTH
   animators emit.

2. **`web/kiri/swarf-lightstream.js` is untracked.** It was introduced
   in the prev session as an overlay that listened to swarf.tool.move
   and drew a ribbon. It is not in git HEAD. Do not treat it as
   canonical via git blame — it has no blame. The file, and
   `src/kiri/app/swarf-line-ext.js` (exposes window.__swarfLine), were
   both born uncommitted.

Final ribbon architecture (what ships)
──────────────────────────────────────
`web/kiri/swarf-lightstream.js` — event-driven, three-layer flat quad
ribbons, additive-blended:

  • Listens to `swarf.tool.move` (fired by anim-2d + anim-3d on every
    mesh_move with a non-zero id).
  • Appends points to a growing polyline. Breaks only on genuine
    rapids/retracts: XY > 50 mm OR Z > 8 mm between frames.
  • `buildQuads(width)` walks the polylines and generates horizontal
    quad vertices per segment. Tangent T = B−A, lateral normal
    N = rotate90(T) in XY → quads lay flat on the cut plane, thin in Z.
  • Three stacked Mesh objects (MeshBasicMaterial, additive, fog: false,
    depthTest: false, DoubleSide), widths and opacities chosen so
    accumulated brightness is highest at the centerline and falls off
    smoothly to the edges:

        core:  0.4 mm × 0.58 alpha × #ff2a1a  (tight bright centerline)
        halo:  1.4 mm × 0.18 alpha × #ff8060  (mid glow)
        bloom: 4.2 mm × 0.06 alpha × #ff6040  (wide diffuse bloom / soft edge)

  • Visibility controlled by `window.__swarfLightstream` + the
    `swarf.lightstream.toggle` event fired by swarf-sim-bar.js's pill.
    Cleared on `animate` / `function.animate` / `swarf.clear`.

DO NOT walk the scene graph to harvest line meshes. The afternoon
burned on that — grid, axes, build-volume cage, and ruler marks all
got harvested as "toolpaths" in various failure modes. If the next
session wants to pre-populate the full toolpath ribbon at the moment
TOOLPATHS finishes computing (instead of only drawing during SIMULATE
play), the correct data source is the worker's `print.output` array
(see `src/kiri/mode/cam/work/anim-2d.js:25` / `anim-3d.js:45` where
`path = print.output.flat()`). Hook a worker dispatch that ships the
flattened path to the main thread on preview.end and feed it through
the same buildQuads pipeline. Do NOT add scene.traverse() back in.
